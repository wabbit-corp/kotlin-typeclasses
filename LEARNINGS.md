# Learnings

- All-open and no-arg are a good fit for the current raw compiler harness. Unlike KSP/KAPT, they only need extra compiler plugin jars plus repeated annotation options, not a generated-sources pipeline.
- `TypeId` needs more than star-projection preservation. Use-site variance (`in`/`out`) has to survive FIR capture, planner unification, IR reconstruction, and canonical-name rendering, otherwise `TypeId<Array<Int>>` and `TypeId<Array<out Int>>` collapse incorrectly.
- Reified `TypeId<T>` cannot be implemented with compiler-baked canonical strings alone because the semantic identity only becomes concrete at the instantiation site. The reliable shape is `typeOf<T>()` in IR plus a runtime `typeId(KType)` factory.
- `Nullable`, `NotNullable`, and `TypeId` fit the same builtin-proof pattern as the earlier utility proofs: planner-visible synthetic rules plus a final IR validity check. The opt-in-only carrier objects can stay internal as long as the public proof interfaces remain opt-in-free and the compiler materializes the carrier singletons on behalf of user code.
- Marking the singleton proof carriers with `@InternalTypeclassApi` is fine as long as the public proof interfaces and helper methods stay unannotated. End users can summon `Same`, `Subtype`, `KnownType`, and the other builtin proofs without opting in because their source never references the carrier singletons directly.
- Some third-party interop failures are just invalid fixtures, not compiler-plugin conflicts. AtomicFU's `updateAndGet` is a top-level extension in `kotlinx.atomicfu`, so tests need to import it explicitly instead of assuming it is a member on `AtomicInt`.
- AtomicFU interoperability tests also need to respect AtomicFU's own IR constraints: atomic operations must target direct atomic properties, not atomics leaked through parameters or locals. A failing `updateAndGet` on a function parameter is an AtomicFU misuse, not evidence that contextual rewriting broke.
- Some interoperability tests need compile-time platform stubs rather than published runtime dependencies. Parcelize is the first concrete example: the harness needs the compiler plugin and parcelize runtime jar, but `android.os.Parcelable` itself can be satisfied by tiny synthetic test sources.
- Builtin rules that are defined in terms of other builtins still need their own IR validation path if prerequisite success is only approximated during planning. `StrictSubtype` exposed this because planner success for builtin `NotSame` does not imply the later IR check will succeed.
- The clean runtime shape for these proof builtins is still zero-state singletons. The interesting API lives on the proof interfaces (`coerce`, `flip`, `compose`, `andThen`, `toSubtype`, `toNotSame`, `bracket`), while the compiler continues to materialize a single shared witness object.
- Interop harnesses need a first-class way to express compiler plugin ids and repeated `-P` options, not just extra jars. Power Assert is a good example: it has no meaningful runtime dependency for these tests, but it does require repeated `function=<fqName>` plugin options.
- FIR call refinement should only hide typeclass context parameters on actual planner success. If ambiguous or recursive evidence is treated as “good enough”, the frontend lies on behalf of the backend and later IR-only diagnostics become much harder to reason about.
- `@Derive` bookkeeping must preserve which typeclass head a class derives, not just that the class had some derive annotation. Collapsing derivation facts to “class X is derivable” makes unrelated heads appear satisfiable during FIR masking.
- `KnownType<T>` materialization does not require `kotlin-reflect`, but `KType.toString()` is less stable without it. Tests that care about proof availability should avoid asserting an exact Kotlin-reflect string rendering unless the harness explicitly adds `kotlin-reflect` at runtime.
- Rule legality has to be enforced at indexing time, not only by FIR declaration checkers. If scanners still index invalid member- or nested-scope `@Instance` declarations, they can manufacture bogus ambiguity at unrelated call sites before the declaration error even becomes useful to the user.
- Some review regressions are not expressible as ordinary source tests because Kotlin rejects them before the plugin path is reached. The attempted “mismatched contextual order” regression failed this way because `Ord<Int> : Eq<Int>` already makes the plain Kotlin context search ambiguous.
- Sealed derivation for a non-generic root cannot safely absorb subclass-only type parameters. If a subclass case type still references variables outside the root’s quantified variables, the derivation must be rejected instead of silently dropping the case or emitting an unusable derived rule.
- Interop test harnesses should declare required third-party compiler ecosystems explicitly. A `requiredPlugins = listOf(...)` model is more durable than sniffing source text for annotations or imports, and it scales to plugin-specific jars, flags, and future options.
- Compose interop is compatible with keeping the generic JVM test floor at `1.8` as long as the harness lets individual support plugins request a higher minimum target. The mistake was the global `21`, not the existence of a higher target for Compose-specific tests.
- Support-artifact discovery should not assume the current test JVM already loaded every third-party jar. Looking in the Gradle dependency cache is a practical fallback for compiler-plugin interoperability tests.
- Builtin proof surfaces are cleaner when they reuse the same rule planner as ordinary typeclass instances and defer the final validity check to IR materialization. `Same`, `NotSame`, `Subtype`, `KnownType`, `SameTypeConstructor`, and `IsTypeclassInstance` all fit that shape without special-casing `summon`.
- `Subtype` proofs should lean on Kotlin’s own IR type checker, not an ad hoc supertype walk. The compiler checker gets nullability, declaration-site variance, and star projections right, and it immediately exposed that `Contravariant<Int> <: Contravariant<Any>` is false because contravariance reverses the direction.
- `KnownType<T>` needs a runtime factory boundary instead of constructing proof implementations directly in the compiler plugin. Synthesizing a call to `knownType(typeOf<T>())`-style runtime API keeps the library surface in charge of the evidence representation.
- Supporting `Covariant<*>`-style proof goals required first-class star projections in the internal `TcType` model. Treating `*` as “missing type information” was enough for earlier phases, but builtin proof goals need to preserve it all the way from FIR/IR type capture to rendered diagnostics.
- PHASE-style regression tests still need ordinary Kotlin validity checks. One PHASE12 “plugin failure” was really `data class Invariant<A>()`, which never reached plugin logic because Kotlin rejects parameterless data classes first.
- The integration harness should model third-party compiler/runtime ecosystems as pluggable support bundles, not one-off conditionals. `kotlinx.serialization` needed both an extra runtime jar and an extra compiler plugin jar, and future interoperability suites will need the same shape.
- Builtin `KSerializer<T>` evidence works best when it delegates to `kotlinx.serialization.serializer<T>()` instead of trying to replicate serializer lookup in the typeclass plugin. The missing piece is a separate FIR/IR gate that only offers the builtin rule for types that are plausibly serializable from metadata, standard-library rules, or still-generic placeholders.
- `TcType` classifier ids are stored in `ClassId.asString()` form, which is slash-based. Any whitelist keyed by classifier id must normalize to that format; dotted FQNs silently misclassify builtin types like `kotlin.Int`.
- K2 IR can present supplied arguments as a dense prefix with trailing `null` placeholders for omitted contextual/default slots. Rewriting needs a real matcher that can choose whether a given prefix argument is preserved contextual evidence or the first regular argument.
- Raw `IrExpression.type` is not always the apparent type a user wrote. Generic helper calls such as `summon<Debug<Int>>()` still expose their return type as the callee's type parameter `T`, so type inference must sometimes recover the instantiated type from the call's explicit type arguments.
- Default methods on generic typeclass interfaces reuse the owner class's type parameters, not just method type parameters. Rewriting `Debug<A>.debug` therefore needs owner-type substitution before it can synthesize extra contextual prerequisites like `Show<A>`.
- Kotlin lowers `receiver[index] = value` contextual `set` calls with `origin=EQ`, a real `extensionReceiver` that actually holds the index expression, the real receiver as the first raw value argument, and a synthetic trailing `null` shim. Rewriters need to swap receiver/index and drop that trailing `null`.
- Kotlin lowers `needle in haystack` contextual `contains` calls with `origin=IN` and the operands in the opposite slots from the original extension signature. Rewriters need to swap the synthetic extension receiver and first raw value argument back before rebuilding the contextual call.
- Kotlin 2.3.10 exposes call-refinement hooks for functions but not for property access expressions. Contextual property getter declarations can still compile, but the read site fails with `no context argument ... found` because FIR never offers a property-access refinement hook analogous to function-call refinement.
- A function-only wrapper does not unblock contextual property reads. `println(intLabel)` against a generated `fun intLabel(): String` still fails with `function invocation 'intLabel()' expected`. The only plausible plugin-side escape hatch is a property wrapper strategy: hide the original contextual property, generate a no-context wrapper property for resolution, and let IR rewrite wrapper getter calls back to the original getter with synthesized typeclass evidence.
- Some test failures are simply invalid fixtures. The `rewritesContextualContainsOperatorCalls` PHASE9 case currently reads a `private` property from outside its declaring class, so it fails before the plugin is involved.
- Recursive derived metadata is safer when it stores stable runtime slots rather than provider lambdas or nested derived-instance expressions. In practice that means emitting `RecursiveTypeclassInstanceCell` objects into metadata and filling them through the public property setter after the derived instance is created.
- IR wrapper lookup cannot assume `callableId` exists. Local functions, anonymous-object members, and some delegation-generated symbols arrive as late-bound IR functions without a callable id, and the rewriter must simply skip wrapper matching for them instead of crashing.
- K2 IR value-argument indices are not the same as `IrFunction.parameters` indices. For member access, the logical order is `context parameters`, then the extension receiver slot, then regular value parameters, while the dispatch receiver is stored separately.
- Receiver-only generic inference is not reliable if the plugin only reconstructs type arguments from explicit type arguments and local typeclass contexts. IR rewriting also needs to infer from dispatch receivers, extension receivers, and surviving value arguments.
- Member helper accessors like `context(t: Tc<A, B>) fun <A, B> customSummon(): Tc<A, B> = t` need dedicated regression coverage. Top-level helper coverage is not enough because dispatch receivers change the IR slot layout.
- Extension operators with unrelated regular argument types, such as `context(Index<A>) operator fun <A> A.get(index: Int)`, are a good canary for receiver-only inference bugs.
- Contextual lambdas can preserve already-resolved typeclass context arguments in the IR call's raw value-argument list. Rewriters cannot assume every `@Typeclass` context parameter has been dropped before they reconstruct type arguments or explicit-argument forwarding.
- Anonymous functions reached through higher-order calls are safer to analyze through their IR parent chain than through transformer-local traversal state alone. The parent chain is what exposed the real downstream failures in `cc-plugin-main`.
- `context(value: T) fun <T> summon(): T = value` stops being a typeclass lookup only after substitution. Any IR rewrite that classifies context parameters before substituting generic type arguments will eventually mis-handle `summon<SomeTypeclass<_>>()`.
- FIR-side type inference for contextual calls must unify against the explicit extension receiver as well as value arguments. Operator helpers like `context(Index<A>) operator fun <A> A.get(Int)` do not expose enough information through regular arguments alone.
- Synthetic implicit receivers added during FIR expression resolution can leak into unrelated overload resolution and lambda adaptation. The concrete failure shape here was `run { ... }` being re-resolved as `T.run { ... }` with an unusable synthetic receiver.
- Some invalid `@Instance` declarations are silently ignored by resolution if the plugin only indexes “good” shapes. That is not enough. Declaration-site FIR checkers are required so bad instance declarations fail the build instead of merely becoming inert.
- Mutual prerequisite cycles such as `Foo<A> <- Bar<A>` and `Bar<A> <- Foo<A>` need to preserve `Recursive` through the planner. If recursive candidates are discarded and collapsed into `Missing`, user diagnostics become misleading and the regression surface looks like a plain “no context argument found” failure.
- Typeclass hierarchy support has two separate surfaces: global rule extraction and local contextual evidence. Expanding only `@Instance` declarations is insufficient; local `context(group: Group<A>)` evidence must also be expanded to inherited heads such as `Monoid<A>` and `Semigroup<A>` for `summon` and contextual-call refinement to work.
- Hierarchy traversal must track visited nodes per path, not globally. A shared visited set across sibling branches can incorrectly hide a valid typeclass path behind an earlier invalid one in diamond-shaped hierarchies.
- Instance-function validation cannot stop at syntax. Generic rule parameters that appear only in prerequisites make the rule impossible to select coherently from the requested goal, so they need to fail at declaration site instead of surfacing later as planner ambiguity or “missing evidence”.
- Direct self-recursive rules are worth rejecting at declaration time even though broader recursive cycles are still handled by the planner. The user-facing diagnostics are much clearer when `Show<Box<A>> => Show<Box<A>>` is rejected where it is declared.
- Local generic functions do not always have stable `callableId`s in IR. Any cache or type-parameter-owner key that assumes `callableId` exists will eventually collapse unrelated local wrappers together; use a fallback identity derived from source offsets and parents.
- Kotlin 2.3 does not support context parameters on classes or constructors. Regression tests for constructor-delegation rewriting must use plain constructors with contextual calls in the delegated `this(...)` expression, not `context(...) class` or `context(...) constructor`.
- IR rewriting for member methods on generic typeclass interfaces must substitute both the method’s own type parameters and the owner typeclass’s type parameters. Default methods like `Debug<A>.debug` infer `A` from the dispatch receiver, not from method-level type arguments.
- Some integration tests can accidentally combine multiple compiler surfaces. The `derivedInstancesCanUseContextualFieldInstances` regression only started exercising derivation cleanly once the test pinned `Wrapper<Int>` in a local value instead of also relying on generic constructor-call inference through `Wrapper(listOf(...))`.
